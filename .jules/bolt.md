## 2024-06-25 - Replace Multiple Queries with PostgreSQL RPC
**Learning:** This codebase leverages Supabase and PostgreSQL. In some places, multiple queries were being made concurrently using `Promise.all` which, while faster than sequential queries, still requires transferring all intermediate data to the Node.js server and aggregating it there, which increases memory footprint and network serialization costs. Migration scripts showed that a specific RPC (`get_reception_snapshot`) had already been created for exactly this purpose—to do the aggregation directly in the database using JSON functions.
**Action:** When working on Supabase/PostgreSQL projects, always check the migration files (`supabase/migrations/*.sql`) for existing RPC functions designed to perform complex aggregations in a single database round-trip before resorting to multiple client-side queries.## 2026-06-26 - Replaced Promise.all Client-Side Aggregation With Postgres RPC
**Learning:** When fetching metrics or statistics, especially when aggregating large rowsets like 'patients' for a clinic dashboard, check if an existing RPC exists to perform this server-side (like `get_reception_snapshot`). Retrieving all rows with `Promise.all` and counting them in Node.js unnecessarily blocks the event loop, loads redundant data into memory, and increases database roundtrips.
**Action:** Use PostgreSQL aggregate functions or pre-existing RPCs (e.g. from `supabase/migrations/005_performance.sql`) instead of performing basic metrics filtering (like count, avg) on the JS client layer.
